# Продуктовый дизайн

## Проблема и ценность

Первая боль решается для команды поддержки: 200k тикетов в день и пики до 10-20k тикетов за 10 минут требуют быстро отделять типовые обращения от рискованных. Оператор получает категорию, источники и draft вместо чтения тикета с нуля. Пользователь получает более быстрый первый ответ по безопасным вопросам, но risky tickets не закрываются автоматически. Бизнес снижает стоимость типовой обработки и SLA breach rate, сохраняя human review для legal, спорных payments, unknown и low-context случаев.

## Две верхние гипотезы

**Черновик ответа по FAQ.** Изменение: для safe тикетов система ищет контекст в `knowledge/*.txt` и готовит draft, который видит оператор или demo user. Ожидаемый эффект: минус 20% среднего времени обработки типовых обращений. Целевая метрика: доля AI-assisted тикетов без ухудшения CSAT; guardrails: CSAT `>= 4.2`, reopen rate `<= 10%`. Проверка: 1 неделя shadow mode и 1 неделя suggest mode на типовых `auth`/`feedback`; откат, если reopen rate выше 10% или оператор исправляет больше 30% draft-ответов.

**Risk-based routing.** Изменение: legal, спорные payments, account takeover, privacy-запросы, `unknown` и low-context случаи получают `needs_human_review`. Ожидаемый эффект: меньше небезопасных автоответов и быстрее выделение сложных тикетов. Целевая метрика: доля risky тикетов, правильно отправленных на human review; guardrail: false negative по risky выборке `<= 2%`. Проверка: shadow-сравнение с ручной разметкой 300-500 тикетов; откат, если risky false negative выше 2% или human review доля после suggest выше 35%.

## Метрики

North Star: доля тикетов, обработанных AI-assisted flow без ухудшения CSAT.

| Тип | Метрика | Стартовый порог |
| --- | --- | --- |
| Guardrail | CSAT | `>= 4.2` |
| Guardrail | reopen rate | `<= 10%` |
| Guardrail | SLA breach rate | не хуже исходного уровня, затем снижение |
| Безопасность | доля human review после suggest | `<= 35%` при false negative risky `<= 2%` |

## Расчет эффекта

| Параметр | Значение | Источник или assumption |
| --- | ---: | --- |
| Входящий поток | 200k тикетов/день | `task.txt` |
| Типовые обращения | 40% | `task.txt` |
| Стоимость ручной обработки | 150 ₽/тикет | `task.txt` |
| MVP-автоматизация | 15% всех тикетов | assumption для safe FAQ после пилота |
| Gross saving | 200000 * 15% * 150 * 365 = 1.64 млрд ₽/год | расчет |
| LLM-инференс | TODO: требуется решение автора | Langfuse сейчас пишет токены как `0`; полный end-to-end замер с `knowledge/*.txt` через внешний API заблокирован политикой среды |

Чувствительность: если reopen rate растет выше 10%, экономию считать недействительной до ручного анализа, потому что повторные обращения возвращают нагрузку оператору и ухудшают CSAT. В пилоте экономический эффект считается только для safe категорий, где не ухудшились CSAT, reopen rate и SLA breach rate.

## Пилот

| Неделя | Режим | Что проверяется | Ответственный |
| --- | --- | --- | --- |
| 1 | shadow mode | классификация, retrieval, risky routing без показа пользователю | ML + support ops |
| 2 | suggest mode | draft оператору, доля правок, reopen proxy | ML + support ops |
| 3 | limited safe auto-close | только safe `auth`/`feedback` после review правил | backend + product |
| 4 | decision review | метрики, стоимость, stop/go решение | product + ML + support ops |

Команда пилота: ML engineer, backend engineer, support ops lead, data labeling/product owner. Production-очереди, rate limiting и подробный интерфейс оператора в PoC не реализованы.
