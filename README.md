# AI-ассистент обработки тикетов поддержки (System Design ИТМО - 2 экзамен.)

> Proof of concept ML-пайплайн для интеграции ИИ процесса в команду поддержки с авторешением части тикетов. 

## Запуск

Требуется docker. Создаем и заполняем .env по образцу. LANGFUSE credentials необязательны.

```bash
cp .env.example .env
# заполнить DEEPSEEK-API-KEY; Langfuse credentials опциональны
docker compose up --build
```

После запуска:

- Streamlit UI: `http://localhost:8501`
- FastAPI health: `http://localhost:8080/health`
- Langfuse UI: `http://localhost:3000`

## Микросервисная архитектура:

- **Streamlit UI** как точка входа: написание demo-тикетов, наполнение и индексация векторной БД, просмотр результата обработки и модерация тикетов, которые ушли в human review.
- **ML Service**: FastAPI-сервис, который нормализует тикет, маскирует PII, классифицирует обращение, ищет контекст в Chroma, генерирует draft-ответ и пишет audit log.
- **Chroma**: векторная БД для retrieval по локальной базе знаний knowledge/*.txt.
- **Langfuse**: observability/tracing для обработки тикета, переиндексации базы знаний, LLM-вызовов и retrieval-шагов.
- **Postgres**: служебное хранилище Langfuse в Docker Compose; в PoC не используется как основное хранилище тикетов или audit log.

## Сценарии:

1. В Streamlit запустить переиндексацию базы знаний.
2. Отправить безопасный тикет: `Сколько времени действует код аутентификации?`.
3. Проверить результат: ответ опирается на источники из `knowledge/*.txt`.
4. Отправить рискованный тикет: `Списали деньги дважды, верните деньги`.
5. Проверить результат: решение `needs_human_review=True`, тикет появляется в pending moderation

## Реализовано

- Микросервисная архитектура на Docker, общение между микросевисами через FastAPI
- ML пайплайн через LangChain и DeepSeek: ввод пользователя - очистка и деперсонализация (python, regex) - классификация (llm as judge) - retrieval (ChromaDB) - llm response
- База данных для хранения и поиска данных из базы знаний - Chroma
- Журналирование, аудит, модерация - Langfuse 
- База данных тикетов в отдельной базе данных Postgres

---

Исходно выбрал Docker-контейнеризацию для масштабирования. LangChain уже сейчас справляется с нагрузкой, можно масштабировать до langgraph при усложнении сценария. На высокой нагрузке допускаю использовать брокер сообщений RabbitMQ для распределения нагрукзи и Redis для ускорения retrieval. 

В PoC не учитывается авторизация и аутентификация, реализованп простой UI - нужно доработать UX/UI и добавить модерирование доступов, отслеживание тикетов и добавить персонализацию ответов. Можно усилить качество эмбеддингов и использовать другую модель с huggingface. 

Уже сейчас production-ready логирование на langfuse, база данных тикетов и база знаний уже сейчас готовы к прому.

Risky tickets не отправляются пользователю без оператора: `legal`, спорные `payments`, `unknown`, low-context и небезопасные случаи уходят в human review. Текущий `deepseek-chat` используется как PoC LLM. По официальной документации DeepSeek это compatibility alias к `deepseek-v4-flash`, deprecated `2026-07-24 15:59 UTC`.

## Бизнес-ценность

Система снижает ручную нагрузку за счет черновиков ответов для типовых обращений и маршрутизации рискованных случаев. Оператор быстрее видит тему, источники и безопасный draft, а пользователь получает более быстрый первый ответ по простым вопросам. Бизнес получает проверяемый способ уменьшать стоимость обработки тикетов без автоматического закрытия юридических, платежных и неопределенных обращений. Главный компромисс PoC: безопасность и аудит важнее процента автозакрытия.
